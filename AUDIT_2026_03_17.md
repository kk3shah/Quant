## 3. 🔴 Asymmetric Order Types (Structural Edge Leak)

**File**: [engine.py](file:///Users/kushalshah/Documents/Quant/quant_bot/strategies/engine.py)  
**File**: [handler.py](file:///Users/kushalshah/Documents/Quant/quant_bot/execution/handler.py)

### Problem
| Action | Order Type | Fee Rate | Slippage |
|:---|:---|:---|:---|
| **Buy** | `limit` (bid) | 0.16% (maker) | May not fill → missed entries |
| **Sell** (stop loss) | `market` | 0.26% (taker) | Immediate but expensive |
| **Sell** (take profit) | `market` | 0.26% (taker) | Immediate but expensive |

```python
# engine.py line 223, 269, 287: All sells use market orders
self.execution_handler.submit_order(symbol, qty, 'sell', order_type='market', is_strategy_exit=True)
```

**Result**: The bot saves 0.10% on entries (often failing to fill) but **always pays maximum fees on exits**. On a 2% target profit with 0.16% entry + 0.26% exit = 0.42% fee drag. That's 21% of the target profit eaten by fees.

### Worse: Unfilled Limit Buys
When buy limit orders don't fill (price moves away), the bot records no entry — but on the next cycle, it may try again at a higher price. The `entry_price` logic in `get_entry_price()` uses `fetch_my_trades()` which returns the last actual fill, not the intended price. This creates phantom tracking issues.

## 4. 🔴 Entry Price Tracking Is Unreliable

**File**: [handler.py](file:///Users/kushalshah/Documents/Quant/quant_bot/execution/handler.py#L222-L238)

```python
def get_entry_price(self, symbol):
    trades = self.exchange.fetch_my_trades(symbol, limit=50)
    # Simple Last Buy Price Proxy
    last_buy_price = 0
    for t in reversed(trades):
        if t['side'] == 'buy' and t['symbol'] == symbol:
            last_buy_price = t['price']
            break
    return last_buy_price
```

### Problems
1. **Only tracks last buy** — if you DCA'd (bought twice), it ignores the first buy → wrong average entry.
2. **If a sell happened between buys**, the "last buy" might be from a completely different position cycle, giving a stale/wrong entry price.
3. **No local state** — relies entirely on exchange API every cycle → slow, rate-limit-heavy, and fragile.

### Impact
The trailing stop and take-profit logic all depend on `entry_price`. If entry price is wrong, the bot might:
- Sell profitable positions at a "loss" (thinking entry was higher)
- Hold losing positions too long (thinking entry was lower)

## Lessons for Feedback Learning

> [!CAUTION]
> **Never deploy a strategy without backtesting it first.** The current bot went live with untested parameters and has been bleeding capital since day one.

### Key Principles to Encode
1. **Fee math comes first**: If your average trade profit doesn't exceed 3x your round-trip fees, the strategy is unprofitable by design.
2. **Don't chase pumps**: Sorting by past returns guarantees buying at the top.
3. **Symmetric execution**: If you use market orders for exits, you must account for taker fees in your profit targets.
4. **Track everything locally**: Relying on exchange API for entry prices is fragile and slow.
5. **Narrow timeframe = narrow targets**: 15m candles rarely produce 10% moves. Match your TP/SL to the timeframe.
6. **Win rate vs risk:reward**: A 7% SL with 10% TP needs >58% win rate. Know your numbers before trading.
